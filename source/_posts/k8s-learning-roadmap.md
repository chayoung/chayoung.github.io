## Context

쿠버네티스는 개념만 많이 알고 있어도 실제 운영에 들어가면 금방 막힌다. 반대로 클러스터만 띄워 보면 왜 그 기능이 필요한지 이해가 얕게 남는다. 그래서 학습 순서를 개념 -> 실습 -> 운영 관점으로 나눠서 밟는 편이 낫다.

이 메모는 "쿠버네티스를 어디서부터 어떻게 공부해야 하는가"에 대한 개인 기준 로드맵이다. 목표는 시험 합격이 아니라, 서비스 배포와 장애 대응까지 이어지는 실전 감각을 만드는 데 둔다.

## Notes

### 1. 먼저 컨테이너와 리눅스 감각을 잡는다

쿠버네티스를 보기 전에 도커와 리눅스 기본기가 먼저 있어야 한다. 파드가 죽고 다시 뜨는 장면만 봐서는 문제를 이해하기 어렵다. 최소한 아래는 익숙해야 한다.

- 이미지와 컨테이너의 차이
- `Dockerfile` 작성 방식
- 프로세스, 포트, 환경변수, 볼륨
- 리눅스 파일시스템과 네트워크 기본
- 로그 확인, 프로세스 종료, 포트 점유 확인

이 단계에서 중요한 건 "애플리케이션이 컨테이너 안에서 어떻게 실행되는가"를 몸으로 이해하는 것이다.

### 2. 쿠버네티스 핵심 오브젝트를 얇고 넓게 본다

처음에는 모든 리소스를 깊게 파지 말고, 서로 어떻게 연결되는지만 먼저 본다.

- Pod
- Deployment
- Service
- ConfigMap
- Secret
- Ingress
- Namespace
- Job / CronJob

이 단계에서는 각 리소스의 YAML을 한 번씩 직접 써보고, `kubectl get`, `describe`, `logs` 정도를 자연스럽게 쓰는 수준이 목표다.

### 3. 로컬 클러스터를 직접 다뤄본다

`kind`, `minikube`, `k3d` 중 하나로 로컬 클러스터를 띄운다. 중요한 건 도구 선택보다 반복 횟수다.

추천 실습:

- 간단한 웹 서버를 Deployment로 배포
- Service로 외부 노출
- 환경변수를 ConfigMap과 Secret으로 분리
- 이미지 태그를 바꾸며 롤링 업데이트 확인
- 일부러 잘못된 설정을 넣고 실패 로그 읽기

이 단계에서는 "왜 안 뜨는지 찾는 습관"이 가장 중요하다. 성공보다 실패 케이스가 더 많이 남는다.

### 4. 네트워크와 트래픽 흐름을 이해한다

쿠버네티스에서 막히는 지점 대부분은 네트워크다. 특히 백엔드 개발자라면 이 구간을 넘겨서는 안 된다.

봐야 할 것:

- Pod IP와 Service IP 차이
- ClusterIP / NodePort / LoadBalancer
- Ingress가 요청을 어떻게 전달하는지
- DNS 이름으로 서비스가 붙는 방식
- readiness / liveness probe가 트래픽에 미치는 영향

실습하면서 "브라우저 요청이 어디를 거쳐 파드까지 가는가"를 그림으로 설명할 수 있으면 다음 단계로 넘어가도 된다.

### 5. 스토리지와 상태 저장 워크로드를 본다

처음에는 stateless 서비스만 다뤄도 되지만, 결국 스토리지를 피할 수 없다.

- Volume, PersistentVolume, PersistentVolumeClaim
- StatefulSet이 필요한 상황
- 데이터가 재시작 이후에도 남는 구조
- 백업과 복구를 어디서 담당할지

이 구간부터는 단순히 YAML을 작성하는 수준을 넘어, 운영 책임이 어디까지인지 생각해야 한다.

### 6. 관측성과 디버깅 루틴을 만든다

쿠버네티스를 배운다고 했을 때 많은 사람이 배포까지만 본다. 실제로는 장애 상황에서 무엇을 먼저 확인할지가 훨씬 중요하다.

기본 루틴 예시:

1. 파드 상태 확인
2. 이벤트 확인
3. 애플리케이션 로그 확인
4. probe 실패 여부 확인
5. 서비스 엔드포인트 확인
6. 인그레스/네트워크 경로 확인

여기에 메트릭, 로그 수집, 알림 도구가 붙기 시작하면 비로소 운영에 가까워진다.

### 7. 실제 배포 방식과 팀 규칙을 배운다

학습 후반부에는 쿠버네티스 자체보다 "팀이 어떻게 쓸 것인가"가 더 중요해진다.

- Helm 또는 Kustomize
- 공통 매니페스트 구조
- 환경별 설정 분리
- CI/CD에서 이미지 빌드와 배포 연결
- 롤백 전략
- 시크릿 관리 정책

이 단계에서는 기술보다 합의가 중요하다. 팀이 제각각 쓰기 시작하면 오히려 운영 난이도가 더 올라간다.

### 8. 관리형 쿠버네티스와 비용까지 본다

로컬 실습만으로 끝내지 말고, EKS/GKE/AKS 같은 관리형 환경이 무엇을 대신해 주고 무엇은 여전히 직접 책임져야 하는지 보는 편이 좋다.

특히 아래 질문에 답할 수 있어야 한다.

- 클러스터 비용은 어디서 많이 나오는가
- 노드 수와 리소스 요청값이 비용에 어떤 영향을 주는가
- 운영팀 없이도 감당 가능한 구조인가
- 지금 정말 쿠버네티스가 필요한가

이 질문을 피하지 않는 것이 중요하다. 도입 판단도 결국 실력의 일부다.

## Suggested Learning Order

1. Docker / Linux 기본기
2. Pod / Deployment / Service 이해
3. 로컬 클러스터 반복 실습
4. 네트워크와 Ingress
5. 스토리지와 StatefulSet
6. 디버깅 루틴과 관측성
7. Helm / CI-CD / 팀 규칙
8. 관리형 쿠버네티스와 비용
