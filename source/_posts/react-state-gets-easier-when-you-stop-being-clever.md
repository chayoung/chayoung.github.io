React 코드를 오래 보다 보면 이상하게 비슷한 냄새가 난다. 상태가 많아서 복잡한 게 아니라, 한 번에 너무 많이 해결하려고 해서 복잡한 경우가 많다. 전역 상태 하나에 폼 값, API 응답, 모달 열림 여부, 임시 필터까지 전부 몰아넣는 식이다.

내가 요즘 지키는 기준은 단순하다. 서버에서 다시 받아올 수 있는 값은 서버 상태로 보고, 화면 안에서만 잠깐 필요한 값은 지역 상태로 둔다. 이 선만 지켜도 코드가 많이 가벼워진다. 검색 입력값과 선택된 탭 정도는 컴포넌트에 남겨두고, 목록 데이터나 캐시 무효화는 별도 레이어에서 다루는 편이 덜 아프다.

컴포넌트를 쪼개는 기준도 마찬가지다. 재사용을 너무 빨리 시도하면 props만 복잡해진다. 아직 두 군데에서만 쓰는 UI라면, 억지로 공용 컴포넌트로 올리지 않는 편이 낫다. 나중에 진짜 공통점이 보일 때 정리해도 늦지 않다.

React가 편한 이유는 작은 단위로 다시 그릴 수 있기 때문이다. 그런데 상태 설계를 욕심내기 시작하면 그 장점을 스스로 죽이게 된다. 대단한 패턴보다, 어디에 어떤 상태를 두는지가 결국 유지보수를 결정한다.
